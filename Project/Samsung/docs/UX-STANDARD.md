# 솔루션 UX 표준

> **목적**  
> 이 솔루션을 다른 고객사에 구축하거나, Cursor/Claude 등 AI에 명령을 넣어 기능을 만들 때 **항상 같은 UX와 품질**이 나오도록 하기 위한 단일 기준 문서입니다.  
> UI(색상·폰트·여백 등)는 제외하고, **동작·패턴·상태·일관성**만 정의합니다.

---

## 1. 적용 범위

- 이 표준은 **이 솔루션 전체**에 적용됩니다.
- 새 페이지·새 기능 추가, 고객사별 커스터마이징, AI로 코드 생성/수정 시 **반드시 이 문서를 따릅니다**.
- Cursor 등에서 작업할 때는 이 문서(`docs/UX-STANDARD.md`)를 컨텍스트에 포함하고, “기존 패턴 따라라”라고 지시할 때 이 문서를 기준으로 합니다.

---

## 2. 원칙 (항상 유지할 것)

| 원칙 | 설명 |
|------|------|
| **일관성** | 동일한 유형의 기능은 화면/고객사와 관계없이 같은 방식으로 동작한다. (예: 목록 필터는 항상 “필터 바 + 패널” 패턴) |
| **예측 가능성** | 사용자가 클릭·선택했을 때 무엇이 열리고/닫히고/비활성화되는지, 다른 화면과 비교해 추측 가능해야 한다. |
| **조건부 동작 명확화** | “어떤 조건에서만 활성/비활성”인지는 코드와 문구로 명확히 하고, 비활성일 때는 클릭 불가·커서·시각 구분을 준다. |
| **피드백** | 토글·필터 적용·패널 열기/닫기 등 상태 변경 시 즉시 화면에 반영되고, 필요 시 포커스·aria로 전달한다. |

## 3. 패턴별 UX 규칙

아래는 이 솔루션에서 이미 정립된 패턴입니다. **새로 만드는 기능도 같은 패턴을 따릅니다.**

### 3.1 필터 (목록 상단)

- **구성**
  - **필터 바**: [Filter 아이콘 + 라벨 + 적용 중인 필터 개수 배지] + [적용된 조건 칩들] + [필터 초기화 버튼]
  - **필터 패널**: 트리거 클릭 시 열리는 패널. 라벨(고정 폭) + 입력/셀렉트/라디오. 푸터에 [필터 초기화] [취소] [필터 적용]
- **동작**
  - 트리거 클릭 → 패널 토글. 패널 밖/다른 트리거 클릭 시 패널 닫기.
  - “필터 적용” 클릭 → 현재 패널 값으로 칩/배지 갱신 후 패널 닫기.
  - “필터 초기화” → 패널 값 초기화 + 칩/배지 초기화.
- **조건부 비활성**
  - 특정 필드 값에 따라 다른 필드를 **비활성화**할 수 있다. (예: “점검 유형 = 권역/사업장별”이면 “사업장/권역” 필드 비활성)
  - 비활성 필드는 `disabled` 및 시각 처리(커서·opacity 등)로 클릭 불가를 분명히 한다.
- **일관성**
  - 자산현황·점검계획 등 **목록이 있는 화면**은 모두 이 “필터 바 + 패널” 패턴을 사용한다. 라벨/옵션만 화면별로 다르게 둔다.

### 3.2 테이블 (목록)

- **행 수 선택**
  - 10/25/50 등 행 개수 선택이 있으면, 선택 값에 따라 **실제로 보이는 행 수가 변경**되고, 페이지네이션 정보(예: “전체 N개 중 1–10 표시”)가 갱신된다.
- **상세 보기 진입**
  - **주요 식별 컬럼**(예: 서버명, 점검명) **셀 클릭** → 해당 행 기준 **상세 슬라이드** 오픈.
  - 또는 **액션 컬럼의 “보기” 버튼** 클릭 → 같은 상세 슬라이드 오픈.
  - 셀 클릭으로 열 때도, 버튼으로 열 때도 **같은 슬라이드·같은 데이터 소스(해당 행)**를 사용한다.
- **조건부 버튼**
  - “보기”처럼 **상태에 따라 활성/비활성**이 갈리는 버튼은, **조건을 코드로 명시**한다. (예: “진행 상태 = 점검 완료”일 때만 활성)
  - 비활성 시: 클릭 불가(`disabled` 또는 `pointer-events` + `cursor` 처리), 시각적으로만 구분(선택 사항).
- **일관성**
  - 테이블 툴바·헤더·행 높이·페이지네이션 배치 등은 화면 간에 가능한 한 동일한 구조를 유지한다.

### 3.3 요약 포틀릿 (접기/펼치기)

- **토글**
  - 우측 끝 **토글 버튼** 하나로 **전체 포틀릿 영역** 접기/펼치기.
  - 접힘 상태: 컨테이너에 `.is-collapsed`(또는 정해진 클래스) 적용. 펼침 상태: 해당 클래스 제거.
- **아이콘**
  - 펼침: 위쪽 화살표(예: `expand_less`). 접힘: 아래쪽 화살표(예: `expand_more`). **텍스트/아이콘 이름으로 전환**하고, 접근성용 aria-label도 상태에 맞게 변경.
- **접힌 레이아웃**
  - 접힌 상태에서는 **한 줄 요약** 형태로, [아이콘+제목] | [메인 숫자] | [요약 항목들]이 가로로 배치된다. 세부 라벨은 접힌 상태에서 숨기고, 불릿·숫자만 노출하는 등 **정의된 접힌 레이아웃**을 따른다.
- **전환**
  - 접기/펼치기 시 높이·패딩 등이 **부드럽게 전환**되도록 transition을 적용한다.
- **자산현황 요약 포틀릿** (`#asset-portlet`)
  - 접힌 상태(**`.is-collapsed`**)에서는 [Figma 401-2093](https://www.figma.com/design/rf40459KZuFvCLiBIvtKfu/Samsung?node-id=401-2093)을 참고해 **지역·OS·환경 세 구역이 가로**로 나란히 보이며, 가로 폭 비율은 **3:4:3**(지역:OS:환경)이다. **OS 구역** 항목 박스에서는 접힌 상태일 때 **OS 명 텍스트(라벨)는 숨기고**, **로고·건수·(기타 OS) devices_other 아이콘·chevron**만 한 줄로 보인다. **기타 OS 아이콘은 접힌 때만** 노출하고, 펼침에서는 다른 OS와 같이 **로고 이미지 없이** 라벨만 둔다. **OS 구역 건수**는 펼침·접힘 모두 **우측 정렬**(접힘 시 flex에서 `margin-left: auto`). 지역·환경 구역은 기존처럼 라벨+숫자 가로 정렬. 구역 컨테이너는 **높이 auto**, **별도 배경·보더 없이** 디바이더로만 구분한다. 토글·아이콘(`expand_less` / `expand_more`)·`aria-label`은 점검계획 포틀릿과 동일 규칙.

### 3.4 상세 슬라이드 (오프캔버스)

- **진입 (1단)**
  - 테이블의 **정해진 컬럼(예: 점검명, 서버명) 셀 클릭** 또는 **액션 컬럼 버튼 클릭**으로 연다.
  - 열 때 **클릭된 행(row)**을 인자로 넘겨, 그 행의 셀 값으로 슬라이드 내용을 채운다.

- **2중(중첩) 슬라이드 — 명시**
  - 슬라이드 **내부 컨텐츠**의 특정 요소(링크·버튼 등)를 클릭하면, **추가 슬라이드 패널이 열릴 수 있다.** 즉 **2단까지(그 이상은 정책으로 제한 가능)** 같은 오프캔버스 패턴을 쌓을 수 있다.
  - **2단 슬라이드**도 **1단과 동일한 UX 구조**를 따른다. (한쪽에서 슬라이드 인, 헤더·닫기, 필요 시 오버레이/포커스 관리 등 **동일한 상호작용 원칙**. 단, **시각적 층위( z-index )**만 한 단계 위에 두어 겹침이 명확해야 한다.)
  - **고객사 구축·AI 구현 시**에도 “1단만 있다”고 가정하지 말고, **2중 슬라이드가 있을 수 있음**을 전제로 닫기 순서·포커스·aria를 설계한다.

- **닫기**
  - 각 단마다 **닫기 컨트롤**을 둔다. 바깥 영역 클릭·오버레이 규칙은 화면 정책에 맞추되, **가능하면 바깥 클릭 시 안쪽(2단)부터 바깥(1단) 순으로 한 단계씩 닫는다** 등 **일관된 스택 순서**를 유지한다.
  - 필요 시 ESC 등 키보드로도 닫을 수 있게 한다.

- **일관성**
  - “목록 + 상세”가 있는 화면은 **같은 슬라이드 패턴**(한쪽에서 슬라이드 인, 오버레이, 닫기)을 사용한다. **상세 안에서 또 상세가 필요한 경우**에도 **같은 패턴을 재사용**하고, 내용·트리거만 화면별로 정의한다.

### 3.5 GNB(상단 헤더 영역)

- **구역 분리**
  - 헤더(GNB)는 **논리적으로 두 영역**으로 나눈다.
    - **좌측**: 현재 체류 중인 **페이지 제목**(및 구역에 맞는 **아이콘**).
    - **우측**: **날짜**(또는 날짜·요일 표기) + 구분선 + **매뉴얼** + 구분선 + **프로필**.
  - 좌·우 영역 모두 **한 줄(가로) 정렬**로 유지하고, 화면별로 다른 요소를 우측에 덧붙이지 않는다. (공통 GNB 구조 유지)

- **페이지 제목·아이콘**
  - 페이지 제목 **앞**에, 해당 화면 성격에 맞는 **아이콘**을 둔다. (예: 대시보드·점검·자산 등 동일 역할의 페이지는 **동일 아이콘 규칙**을 적용해, 고객사 구축 시에도 일관되게 매핑한다.)
  - 제목 텍스트는 **`meta name="gnb-title"`(또는 동일 역할의 단일 소스)**에서 읽어 반영한다. 각 HTML별로 제목만 바꾸고, **갱신 로직은 한 곳**에 둔다.

- **날짜**
  - 우측 첫 블록에 **현재 날짜(및 요일 등)**를 표시한다. 형식은 솔루션 전역에서 **동일 규칙**을 따른다. (고객사별로 로케일만 조정 가능)

- **매뉴얼·프로필(드롭다운/팝오버)**
  - **매뉴얼(Manual)** 버튼과 **프로필** 버튼은 각각 **클릭 시 전용 패널**(드롭다운·팝오버 등)이 **열린다**.
  - **매뉴얼용 패널**과 **프로필용 패널은 서로 다른 내용·항목**을 갖는다. (같은 컴포넌트라도 노출 메뉴·동작은 분리해 정의한다.)
  - 열림 상태에서는 **다른 영역 클릭** 또는 **닫기**로 닫는 등 **동일한 닫기 규칙**을 적용한다. 가능하면 `aria-expanded` 등으로 열림 여부를 전달한다.

- **일관성**
  - 모든 주요 페이지에 **동일한 GNB 마크업·동작**을 사용한다. 고객사별로는 **제목·날짜 로케일·매뉴얼 링크 내용**만 바꾸고, **구조(좌 제목 / 우 날짜·매뉴얼·프로필)**는 유지한다.

### 3.6 LNB(좌측 네비게이션)

- **현재 페이지 반영**
  - **현재 URL(경로)**에 따라 해당 메뉴/그룹이 **active**로 표시된다. 하드코딩하지 않고, 로드 시 또는 라우트 변경 시 **동일 로직**으로 active를 적용한다.
- **이동**
  - 메뉴 클릭 시 해당 페이지(예: `index.html`, `asset-status.html`, `check-plan-management.html`)로 이동한다.

- **하단 토글(화살표) — 메뉴 영역 리사이즈**
  - LNB **하단**에 있는 **이중 화살표 버튼**을 클릭하면, 사이드바가 **넓은 상태 ↔ 좁은 상태**로 전환된다. (폭이 줄어들거나 늘어나는 **리사이즈 UX**; 고객사 구축 시에도 **동일한 토글 위치·동작**을 유지한다.)
  - **펼친(넓은) 상태**: 메뉴 라벨·아이콘이 함께 보이고, 토글 아이콘은 “접기” 방향(예: 좌측을 가리키는 이중 화살표)으로 표시한다.
  - **접힌(좁은) 상태**: 주로 **아이콘만** 보이거나 로고가 축약 표시되고, 토글 아이콘은 “펼치기” 방향(예: 우측을 가리키는 이중 화살표)으로 바꾼다.
  - 토글 시 **본문 영역(#app 등) 너비**도 함께 맞춰져, 좁은 LNB일 때 메인 콘텐츠가 사용 가능한 너비를 확보한다.
  - **접근성**: 토글 버튼에는 `aria-label`(예: “사이드바 접기/펼치기”)을 두어, 화면 전환 의미를 전달한다. 접힘/펼침에 맞게 **라벨·aria 상태**를 갱신할 수 있으면 더 좋다.

- **접힌 상태에서 서브메뉴**
  - 하위 링크가 여러 개인 **메뉴 그룹**은, LNB가 **접힌 상태**일 때 **호버 등으로 옆에 플로팅 메뉴(서브메뉴)**가 나타나 같은 항목에 접근할 수 있게 할 수 있다. (라벨이 숨겨진 상태에서도 동일한 목적지로 이동 가능해야 한다.)
  - 서브메뉴는 포인터가 벗어나면 닫히거나, LNB를 다시 펼칠 때는 닫히는 등 **일관된 닫기 규칙**을 적용한다.

### 3.7 알림/배너

- **닫기**
  - 닫기 버튼이 있으며, 클릭 시 배너가 숨겨지거나 제거된다.
  - 배너가 한 줄일 때 닫기 버튼은 **세로 중앙**에 두는 등, 레이아웃 규칙을 하나 정해 일관 적용한다.

## 4. 품질 기준 (구축·AI 출력 시 체크)

다른 고객사 구축이나 AI로 코드를 만들 때, 아래를 만족하는지 확인합니다.

| 항목 | 기준 |
|------|------|
| **접근성** | 버튼·링크에는 의미 있는 라벨/aria-label. 토글·패널 열림/닫힘은 aria-expanded·aria-hidden 등으로 상태 전달. |
| **클릭 영역** | 클릭 가능한 요소는 버튼·링크·역할이 명확하고, 셀 클릭 시에는 cursor 등으로 클릭 가능함을 표시. |
| **상태 피드백** | 비활성(disabled)은 클릭 불가 + 시각 구분. 로딩·빈 목록·에러가 있으면 해당 상태에 맞는 안내를 표시. |
| **Bootstrap/유틸과 충돌** | 우리가 정한 동작(예: 필터 패널 레이아웃, 포틀릿 접힌 레이아웃)이 Bootstrap 클래스(예: flex-column, justify-content-center)에 덮이지 않도록, **필요 시 구체적인 선택자 또는 !important**로 우리 규칙이 적용되게 한다. |
| **일관성** | 새 페이지/새 목록을 만들 때 기존 “GNB·LNB·필터 바·테이블·상세 슬라이드·포틀릿” 패턴을 재사용하고, 이름만 화면에 맞게 바꾼다. |

## 5. AI(예: Cursor) 활용 시 참조 방법

- **새 기능 추가·수정 시**
  - 이 문서(`docs/UX-STANDARD.md`)를 컨텍스트에 포함한 뒤, “이 솔루션 UX 표준 따라서 구현해라” 또는 “기존 [필터/테이블/슬라이드·2중 슬라이드/포틀릿/GNB/LNB] 패턴이랑 맞춰라”라고 지시한다.
- **고객사별 구축 시**
  - 라벨·옵션·API 연동만 바꾸고, **GNB 구역·제목/날짜/매뉴얼/프로필 동작·필터 구조·테이블 동작·슬라이드 진입/닫기·포틀릿 토글·LNB active·LNB 하단 리사이즈 토글·접힘 시 서브메뉴** 등은 이 문서와 동일하게 유지한다.
- **품질 확인**
  - 구현 후 위 **4. 품질 기준** 항목을 체크리스트로 사용한다.

## 6. 문서 유지

- 새로운 **공통 패턴**이 생기면 이 문서에 섹션을 추가하고, **기존 패턴을 바꿀 때**는 “다른 고객사·기존 화면과의 호환”을 고려한 뒤 수정한다.
